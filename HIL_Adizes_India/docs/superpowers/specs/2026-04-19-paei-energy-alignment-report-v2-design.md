# PAEI Energy Alignment Profile — Report v2 Design Spec

**Date:** 2026-04-19  
**Status:** Approved  
**Scope:** Lambda PDF template redesign only — no backend scoring or payload changes

---

## 1. Context

The existing report ("Adizes Management Style Indicator") is a 9-page PDF generated by an AWS Lambda function (`adizes-pdf-generator`) using Puppeteer + EJS + Chart.js. It describes PAEI management styles at length and uses a radar chart as its primary visual.

The redesign shifts the report's identity and purpose:

| Before | After |
|--------|-------|
| "What is your management style?" | "Where is your energy aligned or misaligned?" |
| AMSI — Adizes Management Style Indicator | PAEI Energy Alignment Profile |
| 9 pages, long PAEI theory sections | 5 pages max, no theory, all insight |
| Radar chart as primary visual | Role card matrix as primary visual |
| External Gap / Internal Gap language | Role Pressure / Energy Tension / Identity Drift |

---

## 2. What Changes, What Stays

### Zero changes to the backend
The Lambda v2 consumes the identical JSON payload as v1:

```json
{
  "assessment_id": "uuid",
  "user_name": "string",
  "completed_at": "ISO timestamp",
  "profile": { "is": "paEI", "should": "PAEI", "want": "pAEI" },
  "scaled_scores": {
    "is":     { "P": 0–50, "A": 0–50, "E": 0–50, "I": 0–50 },
    "should": { "P": 0–50, "A": 0–50, "E": 0–50, "I": 0–50 },
    "want":   { "P": 0–50, "A": 0–50, "E": 0–50, "I": 0–50 }
  },
  "gaps": [ ...existing gap array, retained but not used as primary display... ],
  "interpretation": {
    "dominant_roles": ["A", "E"],
    "style_label": "Administrator",
    "style_tagline": "The Reliable Architect",
    "strengths": "...",
    "blind_spots": "...",
    "working_with_others": "...",
    "combined_description": "The Innovative Organiser — ...",
    "mismanagement_risks": ["Bureaucrat — ...", "Arsonist — ..."]
  }
}
```

`scaled_scores` drives all tension calculations inside the Lambda. No new fields needed.

### Lambda v1 preserved
`lambda/pdf-generator/` is not touched. It remains deployed as `adizes-pdf-generator`.

### Lambda v2 is a new directory and new AWS function
`lambda/pdf-generator-v2/` → deploys as `adizes-pdf-generator-v2`.

### Cutover mechanism
Update the App Runner environment variable:
```
PDF_LAMBDA_FUNCTION_NAME=adizes-pdf-generator-v2
```
Rollback = revert this one env var. No code changes required.

## 3. Tension Calculation Logic (in Lambda v2 JS)

Three tension types are computed per PAEI role from `scaled_scores`:

| Tension Type | Formula | Meaning |
|---|---|---|
| Role Pressure | `abs(should − is)` | Gap between what role demands and current behaviour |
| Energy Tension | `abs(want − should)` | Gap between natural preference and role demand |
| Identity Drift | `abs(want − is)` | Gap between natural preference and current behaviour |

**Tension level classification:**

| Gap size | Level |
|---|---|
| < 5 | 🟢 ALIGNED |
| 5–15 | 🟡 MODERATE |
| > 15 | 🔴 HIGH |

**Per-role primary tension:** the highest of the three tension values for that role.  
**Overall top tensions (Page 1):** the two role+type combinations with the largest absolute gap across all 12 combinations.

---

## 4. Action Path Logic (Page 5, derived in Lambda)

| Section | Source |
|---|---|
| **Stretch** | Role with highest Role Pressure (`abs(should − is)`). "Your role is pulling you here — lean in." |
| **Balance** | Role with highest Identity Drift (`abs(want − is)`). "You are over- or under-expressing this — recalibrate." |
| **Protect** | Role with lowest peak tension (min of the three tension values). "This is working — don't sacrifice it under pressure." |

Each section outputs: role name + color badge + 2-line derived interpretation.

## 5. Five-Page Report Structure

### Page 1 — Energy Alignment Snapshot

**Purpose:** Immediate understanding in under 10 seconds.

**Content:**
- Page header: HILeadership logo · "PAEI Energy Alignment Profile" tagline · red rule
- Name (large) + date + identity line badge: `interpretation.combined_description` text before " — " (e.g. "Innovative Organiser"), or `interpretation.style_label` if no combined description
- **Energy Alignment Matrix:** 4 role cards arranged in a 2×2 grid
  - Each card: role letter (coloured circle) + role name + tension badge
  - IS / SHOULD / WANT mini horizontal bars (same 0–50 scale, role colour at 3 opacities)
  - Tension badge footer: 🔴 HIGH / 🟡 MODERATE / 🟢 ALIGNED
  - Highest-tension role card gets a coloured border highlight
- **Core Insight:** `interpretation.combined_description` (or `style_tagline` if no combined)
- **Top Tensions:** 2 pills showing the two highest role+type combinations

### Page 2 — Tension Map

**Purpose:** Deep read on where pressure lives and what it means.

**Layout:** 4 full role cards, 2 per row.

**Each card contains:**
1. Role header bar (role colour background, white letter + name, tension badge right-aligned)
2. IS / SHOULD / WANT mini bars with numeric values
3. Primary tension highlight box:
   - Tension type label (Role Pressure / Energy Tension / Identity Drift) + signed delta (e.g. +8)
   - 2-line plain-language interpretation — source depends on tension type:
     - **Role Pressure** → `gaps[role].external_message` (maps directly)
     - **Identity Drift** → `gaps[role].internal_message` (maps directly)
     - **Energy Tension** → no pre-built message in payload; Lambda generates it inline using a lookup table keyed on `role + direction` (want > should = "energising pull", want < should = "draining demand")
4. Action cue: 1-line practical nudge derived from the same message source, shortened

Aligned roles (all tensions < 5): card shows bars only + single "Well aligned" line, no highlight box.

### Page 3 — What This Means

**Purpose:** Translate data into lived experience.

**Three sections from `interpretation`:**

| Section heading | Source field |
|---|---|
| When you are at your best | `interpretation.strengths` |
| Where friction shows up | `interpretation.blind_spots` |
| Early warning signs | `interpretation.mismanagement_risks` (each risk styled as a red-left-border alert card) |

Layout: two-column grid (at your best + friction side by side), early warning signs full-width below.

### Page 4 — Style Summary

**Purpose:** Condensed, skimmable style reference. No PAEI theory.

**Content:**
- Core style: `interpretation.style_label` + `interpretation.style_tagline` + `interpretation.combined_description`
- Three columns: **Strengths** · **Watchouts** (`blind_spots`) · **Under stress** (`mismanagement_risks`)
- `working_with_others` removed from report (retained in payload but not rendered in v2)

### Page 5 — Action Path

**Purpose:** Concrete next steps in three directions.

**Three sections (derived algorithmically — see §4):**
- **Stretch →** Role + 2-line interpretation + specific action prompt
- **Balance →** Role + 2-line interpretation + specific action prompt  
- **Protect →** Role + 2-line interpretation + what not to do

Layout: three equal-width cards with left border coloured by role.

## 7. Technical Notes

### No Chart.js
All visuals are HTML `<div>` bars with inline width percentages (`width: ${score/50*100}%`). No canvas, no script injection, no `waitForFunction`. Simpler render path, faster execution.

### Asset sharing
`template/assets/` can symlink or copy from v1. Logo and quadrant SVGs are identical.

### Inlining
`inlineAssets()` logic from v1 is reused verbatim — inlines CSS and converts asset `src=` to base64 data URIs.

### Puppeteer flow (unchanged)
Launch → `setContent()` → `waitUntil: 'domcontentloaded'` → `page.pdf()` → S3 upload → Supabase PATCH.  
No `addScriptTag`, no `page.evaluate()`, no `waitForFunction` — those were only needed for Chart.js.

### Deploy script
Same pattern as v1 `deploy.sh`, with function name `adizes-pdf-generator-v2` and memory/timeout identical to v1.

## 8. Cutover & Rollback

**Cutover:** In AWS App Runner console → Configuration → Environment variables:
```
PDF_LAMBDA_FUNCTION_NAME=adizes-pdf-generator-v2
```
Deploy the service update. All new assessment submissions trigger v2.

**Rollback:** Revert to `PDF_LAMBDA_FUNCTION_NAME=adizes-pdf-generator`. V1 still deployed.

**Side-by-side testing:** Before cutover, invoke v2 directly with a test payload via AWS CLI or console to verify PDF output.
